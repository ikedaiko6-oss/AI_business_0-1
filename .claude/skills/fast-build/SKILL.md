# 高速ビルド手順 (Fast Build)

目的: 「探す→決める→書く→確かめる→片付ける」の順序を固定し、**手戻り(=最大の時間とトークンの無駄)を起こさない。**
コードの中身の質(何を書くか)は [lean-coding](../lean-coding/SKILL.md) を参照。このスキルは「どの順番で動くか」。

## 0. フルフローが要るか先に判断する

- 1行修正・自明なバグ修正・タスクが小さく明確 → 下記を飛ばして直接直す。
- 曖昧・規模が大きい・複数ファイルにまたがる・アーキテクチャに影響 → 1〜5を踏む。
- タスクの大きさに手順の重さを合わせる。小さい作業にフルフローを強制しない。

## 1. 探索 — 書く前に「すでにある形」を見つける

- Grep/Glob で類似実装・既存パターンを先に探す。ゼロから発明しない。あれば流儀に合わせる。
- ファイルを全部読まない。該当箇所だけ読む。
- 独立した探索・調査は同じメッセージで並列に投げる(逐次にしない)。
- 探索に3回以上かかりそうな広い調査は、自分で探し続けず Explore サブエージェントに投げて結果だけ受け取る。

## 2. 設計 — データ形と境界を1回で決める

- 実装に入る前に、データ構造・関数の入出力・責務の境界を決める。ここでの手戻りが最も高くつく。
- 要件が曖昧なら、実装しながら試行錯誤せず、着手前に的を絞った質問を1つ挟むか、規模が大きいタスクは EnterPlanMode で整理してから着手する。
- 決め直しが要りそうな仮の設計のまま実装に進まない。

## 3. 実装 — 一発で最終形を書く

- 下書き→書き直しのループをしない。1で見つけた既存パターンを再利用し、そのまま完成形を書く。
- 新規ファイルの全文生成より Edit による差分適用を優先する。
- 「後で使うかも」のスキャフォールド・TODO・拡張ポイントは書かない(YAGNI、lean-coding §1)。

## 4. 検証 — 最小十分な確認

- 変更に対して一番狭い確認(該当テスト・lint・型チェック)を実行する。フルスイートを反射的に回さない。
- UIが絡むなら実際にブラウザで触って確認する。型チェック・テストは通っても機能が動くとは限らない。
- プロジェクトに verify スキルがあれば使う。

## 5. 片付け — 1コミット = 1意図

- デバッグ出力・死んだコード・使われなくなった分岐を消してからコミットする。
- 整理(リネーム・移動)と機能変更を同じコミットに混ぜない(Tidy First)。
- 変更が大きい・複雑な場合のみ code-review か simplify スキルを1回通す。小さい変更には不要。

## 速度のための追加ルール

- 独立したツール呼び出しは常に並列化する。
- 直前に自分が書いた/編集したファイルを、確認のためだけに読み直さない。
- 作業用の中間メモ・計画ドキュメントを作らない。タスク管理は TaskCreate/TaskUpdate で十分。ユーザーに求められない限り Markdown の計画書を残さない。
- 迷ったら「今このタスクに要るか」で判断する(lean-coding の YAGNI がそのまま手順にも適用される)。
