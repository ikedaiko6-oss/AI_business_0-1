# CLAUDE.md

このリポジトリは AI×事業の 0→1 リサーチ置き場（コードなし、成果物は `research/` 配下の Markdown）。
起点は `research/strength-profile.md`（強みプロファイル）と `README.md`（現時点の結論）。

## 仮説とリサーチの原則（必須・毎回自動適用）

過去のセッションで「太陽光O&Mは競合が来ない」と断定し、後から訂正した経緯がある（`research/2026-07-06-100k-per-day-worldwide-map.md` §9）。
同じ失敗を繰り返さないため、次を毎回・指示なしで実行する。

1. **仮説を提示する前に、その場でリサーチしてから提示する。** 既存の `research/` ファイルで裏付けが取れる場合を除き、WebSearch等で一次情報を取らずに「〜のはず」「確実に」と断定しない。ユーザーに指摘されるのを待たない。
2. **強い断定語（「競合がいない」「需要は確実」「〜が最適」等）を使う前に、反証・競合の実在を積極的に探す。** 都合の良い情報だけ集めない（`S1-competition-analysis.md` の「忖度なし。都合の悪い事実も書く」方針を全リサーチ共通の方針とする）。
3. **数値・相場は出典URL付きで明記する。** 一次情報（省庁・統計・業界団体）を優先し、二次情報・推計しかない場合は「要検証」と明記する。
4. **既存ファイルの結論と矛盾する情報が見つかったら、隠さず訂正として書く。** 優位性・相場は「確定」ではなく「いつまで続くか分からない」前提で書き、再確認の頻度（例: 半年に1回）を明示する。
5. 検証・訂正が終わったら research/ に反映し、コミット・指定ブランチへのpushまで一連で完了させる（都度の許可待ちで作業を止めない）。

## 新しい事業仮説を出す型（毎回この11項目で埋める）

雰囲気で「これ良さそう」と出すのを禁止する。新しい事業仮説は必ず次の型で埋めてから提示する。

| # | 項目 | 中身 |
|---|---|---|
| 1 | 事業仮説 | 一言で何をやるか |
| 2 | 根拠 | なぜ需要があると思うか（一次情報の裏付け） |
| 3 | 強み適合 | 現場力・AI・中京圏・開業済みに合うか（◎○△×）。合わなければ「誰でも言える一般論」なので優先度を下げる |
| 4 | 競合 | 実在する相手を具体名で挙げる。「競合はいない」とは書かない（原則2） |
| 5 | 価格帯 | 相場（出典URL付き） |
| 6 | レッドオーシャン度 | ★1〜5（`S1-competition-analysis.md` の記法に合わせる） |
| 7 | 残っている隙間 | 競合の採算が合わない理由・関係性の堀＝なぜ自分が獲れるか。「確定」ではなく期限付きとして書く（原則4） |
| 8 | 参入コスト | 資格・許認可の壁（無資格でできる部分／提携が要る部分）と必要な自己資金 |
| 9 | 日10万円への道筋 | 単価×件数の算数。**単発の10万円か、ストック化して平均10万円かを明記**（混同しない） |
| 10 | 次に検証すべき質問 | 次の一手（電話・見積もり等）で何を確かめるか。判定を出して終わりにしない |
| 11 | 判定 | S/A/B/C＋理由。ランクの意味は `2026-07-06-100k-per-day-worldwide-map.md` の基準（S=今月着手、A=90日以内、B=6〜12ヶ月、C=やらないと決める）に揃える |
